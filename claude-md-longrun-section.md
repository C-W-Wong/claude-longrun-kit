## Long-run recovery (10h+ pipelines vs 5h usage-limit windows)

Apply whenever launching a workflow/pipeline that may outlive one usage-limit window (>1h of agent work). Triggers are enforced by hooks: PostToolUse(Workflow) reminds about this checklist after every launch; SessionStart surfaces `~/.claude/long-run-state.json` when a pipeline is unfinished; an OS-level launchd/systemd heartbeat covers dead sessions.

0. **Recovery is OPT-IN per pipeline (`autoResume` in its state entry).** Set `autoResume: true` ONLY when the user explicitly asked for unattended/self-healing execution of this run. Default `false`: the circuit breaker still halts cleanly and the SessionStart hook still surfaces the halt, but NOTHING auto-resumes — inform the user and await instruction. User controls anytime: `longrun status | arm | disarm | done | reopen [id]` (see `longrun help`).
1. **Circuit breaker in the workflow script**: any agent that dies twice (usage limit / terminal API error) must HALT the run cleanly at that unit and return partial state — never cascade failures through remaining units. Halt records must name the failed stage (coder vs verifier vs committer).
2. **State directory** `~/.claude/long-run-state/` — one JSON per pipeline, named `<project-basename>-<short-run-id>.json` ({status: running|done, autoResume, project, runId, scriptPath, currentPhase, resumeProtocol, updatedAt}): write your own entry at launch, refresh it at phase boundaries and on every halt/resume, set `status: "done"` when the pipeline finishes (that silences every recovery layer for it, and the heartbeat auto-prunes done entries untouched for 14 days — `CLAUDE_LONGRUN_DONE_RETAIN_DAYS`, or `longrun prune` — so finished runs never accumulate). A legacy single `long-run-state.json` auto-migrates into the directory.
3. **Heartbeats (armed pipelines only)**: when a halt is caused by a usage limit, the error names the reset time (e.g. "resets 4pm") — immediately write it into the pipeline's state entry as `nextResetEpoch` (epoch seconds; plus human-readable `nextResetAt`) and schedule the in-session resume for that moment; clear both fields after resuming. Layer 0 below it: the OS job `com.claude.longrun-heartbeat` fires every 30 min, machine-agnostic — per entry it waits for `nextResetEpoch`, and per pass it revives AT MOST ONE halted armed pipeline (oldest-stalled first; serialized because parallel revivals race the shared usage quota), skipping while any claude session is alive or a fresh recovery session exists and killing a recovery session that made no state progress for 90 min (`tmux attach -t claude-longrun` to watch). In-session timers die with the session; the OS layer is what survives. Multiple armed pipelines simply queue: one revives per pass, the rest follow on later passes.
4. **Working-tree rule on resume**: halted at VERIFY/COMMIT stage → KEEP the tree (the implementer's result is journal-cached; the verifier needs the uncommitted changes). Halted with a CODER dead mid-edit → `git restore` the partial paths to HEAD first. Resume method depends on session: SAME session → `Workflow {scriptPath, resumeFromRunId}` (journal cache, nothing redone). NEW session → the journal is unavailable: determine completed units from the work-order Done/ folder + git log, remove them from the workflow script's unit list, and launch as a fresh run.
5. **Detail lives in project memory** (unit/batch progress, skip lists, environment facts); the state file is a cross-project pointer, not the database.
6. **Portable setup**: the whole mechanism (hooks, scripts, launchd/systemd timer, this section) installs on any machine via the claude-longrun-kit repo (`./install.sh`, idempotent).
