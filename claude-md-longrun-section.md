## Long-run recovery (10h+ pipelines vs 5h usage-limit windows)

Apply whenever launching a workflow/pipeline that may outlive one usage-limit window (>1h of agent work). Triggers are enforced by hooks: PostToolUse(Workflow) reminds about this checklist after every launch; SessionStart surfaces `~/.claude/long-run-state.json` when a pipeline is unfinished; an OS-level launchd/systemd heartbeat covers dead sessions.

0. **Recovery is OPT-IN per pipeline (`autoResume` in the state file).** Set `autoResume: true` ONLY when the user explicitly asked for unattended/self-healing execution of this run. Default `false`: the circuit breaker still halts cleanly and the SessionStart hook still surfaces the halt, but NOTHING auto-resumes — inform the user and await instruction. User controls anytime: `longrun status | arm | disarm | done`.
1. **Circuit breaker in the workflow script**: any agent that dies twice (usage limit / terminal API error) must HALT the run cleanly at that unit and return partial state — never cascade failures through remaining units. Halt records must name the failed stage (coder vs verifier vs committer).
2. **State file** `~/.claude/long-run-state.json` ({status: running|done, autoResume, project, runId, scriptPath, currentPhase, resumeProtocol, updatedAt}): write at launch, refresh at phase boundaries and on every halt/resume, set `status: "done"` when the pipeline finishes (that silences every recovery layer).
3. **Heartbeats (armed pipelines only)**: in-session CronCreate at the usage-limit reset grid +6 min (observed grid 11am/4pm/9pm/2am/7am local → `6 11,16,21,2,7 * * *`); semantics: running → no-op, halted → resume, done → CronDelete itself. Layer 0 below it: the OS launchd job `com.claude.longrun-heartbeat` fires at the same grid and — only when NO claude process is alive AND the state file says running+armed — spawns a recovery session in tmux (`tmux attach -t claude-longrun` to watch). In-session crons die with the session; the OS layer is what survives.
4. **Working-tree rule on resume**: halted at VERIFY/COMMIT stage → KEEP the tree (the implementer's result is journal-cached; the verifier needs the uncommitted changes). Halted with a CODER dead mid-edit → `git restore` the partial paths to HEAD first. Resume method depends on session: SAME session → `Workflow {scriptPath, resumeFromRunId}` (journal cache, nothing redone). NEW session → the journal is unavailable: determine completed units from the work-order Done/ folder + git log, remove them from the workflow script's unit list, and launch as a fresh run.
5. **Detail lives in project memory** (unit/batch progress, skip lists, environment facts); the state file is a cross-project pointer, not the database.
6. **Portable setup**: the whole mechanism (hooks, scripts, launchd/systemd timer, this section) installs on any machine via the claude-longrun-kit repo (`./install.sh`, idempotent).
